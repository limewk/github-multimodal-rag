# 项目进展分析

日期：2026-05-19

## 总体结论

项目目前处于“基础可用 + 检索增强已完成”的阶段。后端索引、Qdrant 入库、仓库级上下文、混合检索、流式问答和最小前端已经形成闭环，适合用于本地演示、课程阶段汇报和继续迭代。

下一阶段的主要矛盾已经从“能否索引和问答”转向“能否稳定回答跨文件结构问题、能否解释复杂代码关系、能否量化 RAG 质量”。因此后续重点应放在代码结构索引、检索调试和评测集上。

## 已完成里程碑

1. 初始化 FastAPI + Vue 工程骨架。
2. 建立 RAG 核心目录：ingestion、processing、retrieval、generation、prompts。
3. 实现仓库索引入口 `/repos/index`。
4. 实现问答入口 `/chat` 和 SSE 流式输出。
5. 接入 Qdrant，并用 `repo_id` 隔离不同仓库。
6. 实现 hash embedding，降低本地开发门槛。
7. 实现仓库文件 manifest 和 overview。
8. 实现 Phase 2 混合检索，改善只靠向量召回导致的覆盖不足。
9. 增加基础测试，保护主要数据流。

## 当前风险和短板

- 前端文案和部分源码中文注释存在编码乱码，会影响演示观感，也可能影响 prompt 可维护性。
- 关键词召回目前是滚动扫描文档，不是倒排索引或 BM25，大仓库下可能变慢。
- 尚未建立结构化代码索引，跨文件调用链、入口流程和模块依赖类问题仍依赖文本片段。
- 图片只作为引用入库，不能理解架构图、截图或流程图中的真实内容。
- 缺少 `/search-debug`、检索分数展示和评测集，调参主要依赖人工观察。
- 当前 API 没有鉴权、任务队列和索引进度推送，更适合本地或受控环境。

## 建议下一步

1. 修复编码乱码。
   - 优先处理 `ui/src/App.vue` 的用户可见文案。
   - 再处理 `src/prompts/rag_prompt.jinja2` 和源码注释。

2. 增加检索调试能力。
   - 新增 `/repos/{repo_id}/search-debug`。
   - 返回向量候选、关键词候选、重排分数、最终上下文和来源元数据。

3. 启动 Phase 3 结构化代码索引。
   - 先覆盖 Python、JavaScript、TypeScript。
   - 先抽取函数、类、导入、入口文件和 API 路由。

4. 建立小型评测集。
   - 选择 1 个真实中型仓库。
   - 准备 10 到 20 个问题，覆盖架构、入口、实现细节、配置、测试和图片引用。

5. 优化前端证据展示。
   - 回答区域展示 Sources。
   - 支持展开命中文件、行号和 source type。

如果用于课程或阶段性汇报，可以将当前项目描述为：

> 项目已完成 GitHub 仓库 RAG 问答的 MVP 和检索增强阶段，具备仓库索引、仓库级上下文、混合检索、流式回答和基础前端。后续将从文本 chunk RAG 进一步升级为结构化代码 RAG，并补齐图片理解、评测和检索调试能力。

# 项目进展分析

日期：2026-06-02

## 总体结论

项目目前处于“基础可用 + 检索增强 + 前端交互增强已完成”的阶段。后端索引、Qdrant 入库、仓库级上下文、混合检索、流式问答、代码结构查询和文件上传已经形成闭环；前端已经从最小问答界面升级为包含仓库预览、代码解析、知识图谱、代码统计、调用图、Treemap 和智能问答的综合界面。

下一阶段的主要矛盾已经从“能否索引和问答”转向“能否稳定解释跨文件结构、能否展示真实调用关系、能否量化 RAG 质量、能否处理图片内容”。因此后续重点应放在结构化代码索引、检索调试、评测集和多模态理解上。

## 已完成里程碑

1. 初始化 FastAPI + Vue 工程骨架。
2. 建立 RAG 核心目录：ingestion、processing、retrieval、generation、prompts。
3. 实现仓库索引入口 `/repos/index`。
4. 实现问答入口 `/chat` 和 SSE 流式输出。
5. 接入 Qdrant，并用 `repo_id` 隔离不同仓库。
6. 实现 hash embedding，降低本地开发门槛。
7. 实现仓库文件 manifest 和 overview。
8. 实现 Phase 2 混合检索，改善只靠向量召回导致的覆盖不足。
9. 增加代码结构接口 `/repos/{repo_id}/structure`。
10. 增加文件上传接口 `/upload/`，支持文本类文件和图片附件。
11. 增强前端：仓库预览、代码解析、多级目录、知识图谱、代码统计、调用图、Treemap。
12. 增强智能问答：LaTeX、打字机输出、历史对话、折叠引用来源、点击引用跳转代码。
13. 增加基础测试，保护主要数据流。

## 当前风险和短板

- 关键词召回目前是滚动扫描文档，不是倒排索引或 BM25，大仓库下可能变慢。
- 代码结构目前偏向 chunk 元数据展示，跨文件调用链、导入关系和真实依赖图仍需更可靠的结构化索引。
- 前端知识图谱和调用图主要基于已有 chunk 元数据和轻量启发式关系，不能完全等同真实静态分析结果。
- 图片只作为引用或附件处理，不能理解架构图、截图或流程图中的真实内容。
- 缺少 `/search-debug`、检索分数展示和评测集，调参主要依赖人工观察。
- 当前认证、存储和索引执行方式更适合本地或受控环境，还不适合直接作为生产系统暴露。

## 建议下一步

1. 增加检索调试能力。
   - 新增 `/repos/{repo_id}/search-debug`。
   - 返回向量候选、关键词候选、重排分数、最终上下文和来源元数据。

2. 启动 Phase 3 结构化代码索引。
   - 先覆盖 Python、JavaScript、TypeScript。
   - 抽取导入关系、导出符号、入口文件、API 路由和函数调用边。
   - 让知识图谱和调用图基于结构化边而不是仅靠启发式匹配。

3. 建立小型评测集。
   - 选择 1 个真实中型仓库。
   - 准备 10 到 20 个问题，覆盖架构、入口、实现细节、配置、测试和图片引用。
   - 记录召回来源、答案质量和失败案例。

4. 增强多模态能力。
   - 为图片附件接入支持视觉输入的模型。
   - 对仓库内架构图、流程图和截图生成 caption 或 OCR 文本后入库。

5. 优化工程化能力。
   - 增加索引进度推送和任务队列。
   - 增加更严格的鉴权和数据隔离。
   - 为前端关键交互补充端到端测试。

## 阶段判定

如果用于课程或阶段性汇报，可以将当前项目描述为：

> 项目已完成 GitHub 仓库 RAG 问答的 MVP、检索增强和交互式可视化前端。系统具备仓库索引、仓库级上下文、混合检索、流式回答、文件上传、历史对话、引用跳转、代码结构展示和多种可视化视图。后续将从文本 chunk RAG 进一步升级为结构化代码 RAG，并补齐图片理解、评测和检索调试能力。
